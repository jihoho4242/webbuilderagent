# Engine Run Contract

`engine-run` is the Manus-inspired, scoped local agentic runtime for WebBuilderAgent.

It does not replace the existing bounded `agent-run` safe patch flow. `safe_patch` stays available for conservative source edits. `agentic_local` adds a staged sandbox workspace where the agent can inspect, edit, run local checks, observe failures, and retry before aiweb validates copy-back.

## Modes

- `safe_patch`: delegates to the existing bounded `agent-run` contract.
- `agentic_local`: stages a filtered project copy, runs the selected sandboxed agent in that workspace, validates changes, and copies back only safe files. Real `agentic_local` cannot use unsandboxed Codex; use `safe_patch` for Codex, `--agent openmanus --sandbox docker|podman`, experimental `--agent openhands --sandbox docker|podman`, experimental `--agent langgraph --sandbox docker|podman`, or experimental `--agent openai_agents_sdk --sandbox docker|podman` for sandboxed autonomy.
- `external_approval`: reserved for elevated capabilities such as package install, external network, deploy, provider CLI, MCP/connectors, or git push.

## Trust Boundary

The agent never receives a writable host project root. aiweb stages a workspace under `.ai-web/tmp/agentic/<run-id>/workspace` and excludes `.env*`, `.git`, dependency folders, generated bulk output, provider credentials, browser profiles, and secret-looking paths.

The copy-back gate validates:

- writable glob envelope
- denylisted and secret-looking paths
- secret-like file content
- binary files
- symlinks and hardlinks
- deletes and high-risk files such as lockfiles or deploy config

Safe source/test/style changes may be copied back automatically. High-risk changes move the run to `waiting_approval`. Unsafe changes move it to `blocked`.

The local backend treats approved real `POST /api/engine/run` and `POST /api/engine/approve` calls as durable background jobs. The HTTP response returns immediately with the selected `run_id`, `.ai-web/runs/<run-id>/job.json`, and the event stream path. The worker then invokes the dedicated `engine-run` bridge with `--run-id <run-id>` so the web console can poll one stable run id from queued through finished. Clients can use either JSON long-poll (`GET /api/project/run-stream`) or typed Server-Sent Events (`GET /api/project/run-events-sse`) for the same append-only event log.

`GET /api/engine/openmanus-readiness` reports Docker/Podman executable availability and local `openmanus:latest` image readiness before the web UI enables approved OpenManus run controls.

Every capability envelope also carries an authz contract and retention/redaction policy. Local engine-run remains project-scoped and token-protected; any SaaS Workbench must authorize `{tenant_id, project_id, user_id}` plus a server-configured role for start, approve, resume, cancel, view, artifact read, and copy-back because `run_id` alone is never authority. The local backend now has opt-in claim-enforced modes: `AIWEB_DAEMON_AUTHZ_MODE=claims` requires `X-Aiweb-Tenant-Id`, `X-Aiweb-Project-Id`, and `X-Aiweb-User-Id`; `AIWEB_DAEMON_AUTHZ_MODE=jwt_hs256` requires `Authorization: Bearer <JWT>` signed with HS256 via `AIWEB_DAEMON_JWT_HS256_SECRET` and carrying tenant/project/user claims (`tenant_id`/`tid`, `project_id`/`pid`, `user_id`/`sub`); `AIWEB_DAEMON_AUTHZ_MODE=jwt_rs256_jwks` requires an RS256 bearer JWT verified against a local JWKS file configured by `AIWEB_DAEMON_JWT_RS256_JWKS_FILE` and fails closed on missing, unknown, or duplicate `kid` entries; and `AIWEB_DAEMON_AUTHZ_MODE=session_token` requires an opaque `Authorization: Bearer <token>` whose SHA-256 hash exists in `AIWEB_DAEMON_SESSION_STORE_FILE` with tenant/project/user claims, expiry, and revocation status. These modes fail closed unless a server-side registry authorizes the requested canonical project root: either the inline `AIWEB_DAEMON_AUTHZ_PROJECTS` allowlist plus optional `AIWEB_DAEMON_TENANT_ID`/`AIWEB_DAEMON_USER_ID` pins, or a local JSON tenant/project membership registry at `AIWEB_DAEMON_AUTHZ_PROJECTS_FILE`. That registry may define tenant members, project members, `roles`, or `user_roles`; invalid role names fail closed instead of defaulting to admin, and route ACLs require `viewer` for reads, `operator` for run/command/cancel/resume actions, and `admin` for approve/copy-back style authority. `X-Aiweb-Project-Id` is treated as a server-configured allowlist identifier, not a client-derivable path hash, and client headers do not grant roles. In claim mode, every allow/deny decision appends redacted audit evidence to `.ai-web/authz/audit.jsonl` using hashed tenant/user ids. Direct artifact reads also classify artifact sensitivity: standard safe artifacts require `viewer`, diffs/logs/sensitive run artifacts require `operator`, and direct approvals JSONL requires `admin`; artifact payloads expose `artifact.acl` so the UI can show why a read was allowed or denied. Unsupported raw authz modes such as `jwt`/`oidc` still fail closed for project-scoped routes instead of falling back to local-token behavior; `jwt_hs256` is a local shared-secret verifier, `jwt_rs256_jwks` is a local public-key/JWKS verifier with no OIDC discovery or network fetch, `session_token` is a local hashed session-store verifier that rejects duplicate token hashes, missing expiry, expired, revoked, and not-yet-valid entries, and hosted OIDC / external SSO discovery remains `not_implemented_fail_closed`. This is local enforcement evidence, not a hosted SaaS identity provider or full external SSO/session service. Events, logs, prompts, screenshots, DOM snapshots, diffs, and command output are retained as project-local run artifacts until the user deletes the run and are redacted at source before display.

`authz-enforcement.json` records the local authorization evidence for each run: project scope, API-token and approval-token route requirements, safe project-path requirement, artifact-reference bounds, per-route permission inputs, role ACL/audit requirements, route-required-role matrix, audit path, local `claims`/`jwt_hs256`/`jwt_rs256_jwks`/`session_token` verifier support, `AIWEB_DAEMON_AUTHZ_PROJECTS_FILE` project-registry support, hashed session-store support, local JWKS-file RS256 support, and the fact that remote SaaS exposure is blocked until `{tenant_id, project_id, user_id}` claims and server-side roles are enforced. This moves local authz from pure metadata into run evidence, while still not claiming remote multi-tenant enforcement exists.

Dry-run writes nothing and returns the planned paths plus a capability envelope, OpenDesign contract hashes, and approval hash. The approval hash is stable across dry-run and real execution for the same goal, mode, agent, sandbox, limits, resume target, copy-back envelope, and OpenDesign contract; the transient run id is excluded from the hash.

The OpenDesign contract records `.ai-web/DESIGN.md`, optional `.ai-web/design-reference-brief.md`, `.ai-web/design-candidates/selected.md`, the selected candidate artifact, and optional `.ai-web/component-map.json` by path, bytes, and SHA-256. UI/source work in Profile D is blocked before agent execution when no selected design candidate or selected candidate artifact is available.

Before copy-back, `design-fidelity.json` records deterministic static fidelity. It blocks source that drops required `data-aiweb-id` hooks from the component map, drifts the selected candidate identity, or leaks forbidden reference/brand terms from the design reference brief. `selected_design_fidelity` must meet the configured threshold before safe changes can be copied back.

When `package.json` exposes `dev` or `preview`, engine-run starts the preview command only through the same sandbox command wrapper and records `preview.json` with command, localhost URL, stdout/stderr excerpt and log paths, process metadata, lifecycle (`persistent_ready`, `exited_after_ready`, timeout/failure variants), readiness status, and teardown status. Long-running preview servers are spawned persistently, probed until a local URL is observed, then stopped after browser observation or on failure. Preview failure prevents copy-back.

When preview is ready, engine-run records desktop, tablet, and mobile Playwright browser evidence in `qa/screenshots.json` plus screenshot files under the run screenshots directory. Placeholder or text masquerading as `.png` is not accepted; screenshot files must pass PNG signature and dimension checks before they are copied into run evidence. The browser observation command runs through the same staged-workspace tool broker/sandbox boundary and must produce screenshot, DOM snapshot, accessibility tree, computed style, interaction-state, keyboard-focus, reversible action/recovery, bounded action-loop, console-error, and network-error evidence. The action/recovery contract exercises safe UI observation steps such as scroll, hover, focus, reversible text-input probe/restore, same-origin anchor click with preview restore, aria-expanded toggle click with Escape recovery, preview URL restore, and external-navigation blocking; form submission, credential entry, file upload, payment/deploy actions, destructive clicks, and non-local navigation are blocked or not performed. Top-level `action_recovery.external_requests_blocked` aggregates blocked non-local requests across viewports, while top-level `action_loop` records the deterministic bounded loop policy, limits, planned/executed/recovery/blocked steps, per-viewport `scenario_plan` / `scenario_results`, `multi_step_evidence`, stop condition, and limitations. Browser observation is localhost-only and feeds the later visual-fidelity verdict loop. `runtime_attestation` records whether preview and browser observation shared the same staged workspace, selected sandbox wrapper, localhost-only policy, and `_aiweb/tool-broker-bin` boundary; it explicitly does not claim a single long-lived container instance or a production open-ended browser agent.

Before the worker acts, engine-run writes `artifacts/project-index.json` and `_aiweb/project-index.json` inside the staged workspace. This dependency-free retrieval context summarizes routes, components, styles, data contracts, auth surfaces, package scripts, test/build/preview commands, and `.env*` filenames without reading environment values.

Before the worker acts, engine-run also writes `artifacts/run-memory.json` and `_aiweb/run-memory.json`. This is a deterministic, bounded lexical memory handoff derived from the OpenDesign contract and project index so the worker can retrieve routes, component hooks, and package scripts without scanning secrets. It truthfully records `retrieval_strategy: bounded_lexical_cards` and `rag_status: not_configured`; it is not an embedding store, LlamaIndex pipeline, or durable cross-run memory yet.

`aiweb engine-scheduler status|tick|daemon|supervisor|monitor` is the project-local durable graph scheduler service surface. It is separate from the monolithic `engine-run` method and records scheduler service decisions under `.ai-web/runs/<run-id>/artifacts/scheduler-service.json` plus append-only `.ai-web/scheduler/ledger.jsonl`. The service derives the next node from the hashed checkpoint graph cursor and `Aiweb::GraphSchedulerRuntime`, records a lease, supported continuation nodes, resume command, graph-node summary, and blockers, and exposes `node_body_executor: engine_run_resume_bridge`. `status` is read-only. `tick` records the scheduler artifact/ledger. `tick --approved --execute` resumes only through the explicit `engine-run --resume <run-id> --approved` bridge and persists `execution_attempted`, `execution_status`, and `execution_result_summary` back into the scheduler-service artifact. `daemon` adds a foreground scheduler loop and records `.ai-web/scheduler/daemon.json`, `.ai-web/scheduler/daemon-heartbeat.json`, `.ai-web/scheduler/worker-pool.json`, `.ai-web/scheduler/leases.json`, and append-only `.ai-web/scheduler/queue-ledger.jsonl`; `--max-ticks 0` is the explicit long-running foreground mode, while bounded `--max-ticks N` is testable and deterministic. Approved daemon execution claims the lease first, then resumes outside the scheduler state lock to avoid nested-lock deadlocks, records `resume_ready_executed`, appends `scheduler.execution.finished`, and marks completed leases inactive after the bridge returns. `supervisor` records `.ai-web/scheduler/supervisor.json` with external systemd/launchd/Windows Task Scheduler templates, heartbeat/lease health checks, restart policy, operator runbook, and explicit `install_performed: false`; aiweb intentionally does not install OS services or start privileged service managers. `monitor` records `.ai-web/scheduler/monitor.json` health evidence over heartbeat freshness, active lease staleness/duplicates, queue-ledger parse status, worker-pool concurrency, and supervisor contract status. The worker pool is a repo-local lease/slot contract over the same resume bridge: runnable work is represented as single-run/start-node lease claims, duplicate active claim keys are blocked on restart, the queue ledger records tick, claim, stale-recovery, and execution-finished events, and worker slots enforce the configured local concurrency cap. It is not a remote queue or distributed cluster. Node bodies still execute through existing engine-run handlers rather than independent distributed workers. This closes the absence of a repo-local foreground scheduler daemon, worker-pool evidence surface, approved inline resume evidence, supervisor/runbook artifact, and repo-local scheduler monitor artifact, but it is still not a production multi-process orchestration cluster.

Before the worker acts, engine-run writes `artifacts/worker-adapter-registry.json` and `_aiweb/worker-adapter-registry.json`. The registry records the selected adapter, whether the selected adapter is executable for the current run, the selected adapter blocking issues, the common `worker-adapter-v1` API, executable adapters (`openmanus` as a sandboxed container worker, `openhands` as an experimental sandboxed headless CLI container worker, `langgraph` as an experimental sandboxed StateGraph bridge, `openai_agents_sdk` as an experimental sandboxed Agents SDK bridge, and `codex` only through safe-patch delegation). Each adapter entry carries explicit `executable`, `execution_blocked`, `blocking_issues`, `sandbox_preflight`, `result_schema`, `driver_readiness`, and `broker_contract` fields; planned-only adapters are never executable until a command driver, sandbox preflight, result schema, broker event flow, and evidence artifacts exist. OpenHands execution is intentionally narrow: aiweb launches a prepared local OpenHands-compatible image with `openhands --headless --json --file /workspace/_aiweb/openhands-task.md`, passes only staged workspace refs, writes `_aiweb/openhands-task.md`, and requires the same docker/podman sandbox preflight and broker evidence as OpenManus. LangGraph execution is also intentionally narrow: aiweb writes `_aiweb/langgraph-worker.py` and `_aiweb/langgraph-task.md`, launches a prepared local Python image with `langgraph` installed, invokes a small `langgraph.graph.StateGraph` bridge over the worker-adapter contract artifacts, and requires `docs/schemas/engine-run-langgraph-result.schema.json`; this is not LangGraph Platform or distributed checkpointing. OpenAI Agents SDK execution is likewise narrow: aiweb writes `_aiweb/openai-agents-worker.py` and `_aiweb/openai-agents-task.md`, launches a prepared local Python image with `openai-agents` installed, imports `agents.Agent`/`Runner`, records SDK orchestration readiness, and requires `docs/schemas/engine-run-openai-agents-sdk-result.schema.json`; default model/network calls are disabled by broker policy (`AIWEB_OPENAI_AGENTS_ALLOW_MODEL_CALL=0`, `AIWEB_NETWORK_ALLOWED=0`). `runtime_broker_enforcement` records that executable adapters must have broker evidence, while MCP/connectors and future adapters fail closed except for known brokered drivers: design-research uses `aiweb.lazyweb.side_effect_broker`, implementation-worker Lazyweb health/search calls use `aiweb.implementation_mcp_broker` only with explicit approval, and `project_files.project_file_metadata`, `project_files.project_file_list`, bounded safe `project_files.project_file_excerpt`, plus bounded literal `project_files.project_file_search` use the same implementation MCP broker as no-network file drivers. The `driver_readiness` object records required/missing artifacts, executable_now, transition_gate, and next_required_evidence so experimental readiness stays machine-readable instead of relying on prose-only caveats. This is an evidence surface for interchangeability and broker readiness, not a claim that all listed workers are production-grade today.

Before the worker acts, engine-run also writes `artifacts/worker-adapter-contract.json` and `_aiweb/worker-adapter-contract.json`. The adapter contract defines the common `prepare/act/observe/cancel/resume/finalize` API, input refs, output manifest fields, and contract violations. Worker output that returns host absolute paths, raw environment payloads, or raw secret-like values blocks copy-back. OpenHands follows `docs/schemas/engine-run-openhands-result.schema.json`, LangGraph follows `docs/schemas/engine-run-langgraph-result.schema.json`, and OpenAI Agents SDK follows `docs/schemas/engine-run-openai-agents-sdk-result.schema.json`; all three must write `_aiweb/engine-result.json` with the expected `adapter`, `structured_events`, `artifact_refs`, `changed_file_manifest`, `proposed_tool_requests`, `risk_notes`, and `blocking_issues`; missing or malformed adapter result fields block copy-back.

`design-verdict.json` is a hard gate. The deterministic local reviewer records hierarchy, spacing, typography, color, originality, mobile polish, brand fit, intent fit, and selected-design fidelity scores. Copy-back is prevented when browser evidence is missing, DOM/a11y/computed-style/interaction/focus/action-recovery/action-loop evidence is incomplete, browser observation records console or network errors, or a required score or average score is below threshold.

`design-fixture.json` records the selected human-approved design brief, golden selected-candidate reference, viewport-specific expected outcomes, allowed variance thresholds, and stored baseline verdict. This gives future eval/backtest runs a stable target instead of relying only on the latest ad-hoc screenshot review.

`eval-benchmark.json` records the repeatable benchmark envelope for the run: task success, build/test pass status, visual fidelity, interaction coverage, action/recovery evidence, bounded browser action-loop evidence, accessibility evidence, repair cycles, approval count, unsafe-action blocking, time-to-pass, and tool/token-cost availability. It carries `human_calibration_status` (`missing`, `seeded`, `calibrated`, or `invalid`) plus the baseline source. When `.ai-web/eval/human-baselines.json` has no matching fixture, engine-run seeds the baseline from the deterministic `design-fixture.json` stored verdict and marks it `seeded`, `human_calibrated: false`. Human baselines live at `.ai-web/eval/human-baselines.json`, follow `engine-run-human-baselines.schema.json`, must use numeric 0..100 scores, must not contain raw secrets or environment values, and are `calibrated` only when positive reviewer evidence exists. `aiweb eval-baseline review-pack [--fixture-id <id>]` writes `.ai-web/eval/human-review-pack.json` from `engine-run-human-review-pack.schema.json` with placeholders, required human-review fields, and an anti-fabrication policy; it never fills reviewer ids, scores, notes, or evidence refs. `aiweb eval-baseline validate --path <candidate>` records redacted validation evidence without importing, while `aiweb eval-baseline import --path <candidate> --approved` copies only a project-local, secret-free, human-calibrated corpus into `.ai-web/eval/human-baselines.json`; all three commands refuse to fabricate reviewer evidence. A calibrated regression gate fails when the current average score is below the calibrated baseline average. Without a valid calibrated human baseline, the regression gate is explicitly marked `evidence_gate_only_no_human_baseline`; it can still fail on missing browser/design/a11y/interaction/action-recovery/action-loop evidence, but it cannot be claimed as a human-calibrated benchmark.

When a design verdict fails and cycles remain, `_aiweb/repair-observation.json` includes preview evidence, screenshot metadata, the design verdict, selected OpenDesign contract identity, and repair instructions. The next sandbox cycle records `design.repair.started` and `design.repair.finished`.

Approval records include an explicit `execute` scope, the run id, capability hash, single-use marker, and the approved capability envelope. Waiting-approval runs also expose structured `approval_requests` explaining why approval is needed, the risk, the exact capability unlocked, affected paths, and provider/package/MCP-specific required fields. Reusing an approval hash after the OpenDesign contract changes is rejected before run artifacts are written.

`supply-chain-gate.json` records the package-install approval gate. For pending package installs it requires an isolated clean cache (`_aiweb/package-cache`), exact package-manager command, registry/network allowlist, lifecycle-script policy, dependency and lockfile diffs, SBOM output, package audit output, and a vulnerability copy-back gate that blocks critical/high findings unless an explicit exception and rollback plan are approved. While the elevated install is still pending, engine-run writes truthful `artifacts/sbom.json` and `artifacts/package-audit.json` evidence with `status: not_executed_pending_approval`; these files prove the SBOM/audit step did not run in the default sandbox profile and cannot be interpreted as a clean SBOM or audit. Package manifest or lockfile changes without a supply-chain request are recorded as blocked gate evidence.

The approved `aiweb setup --install --approved` package-manager path uses the shared side-effect broker evidence helper as `aiweb.setup.side_effect_broker`. Dry-run and missing-approval paths plan the broker but write no broker log. Approved runs write `.ai-web/runs/<setup-run>/side-effect-broker.jsonl` with `tool.requested -> policy.decision -> tool.started -> tool.finished|tool.blocked`, redact stdout/stderr before persistence, mirror events into `setup.json`, and preserve the setup contract that install never builds, previews, runs QA, or deploys. The approved setup command is exact and includes a run-local pnpm store directory (`pnpm install --ignore-scripts --registry https://registry.npmjs.org/ --store-dir .ai-web/runs/<setup-run>/package-cache`). Approved setup runs also write `.ai-web/runs/<setup-run>/artifacts/supply-chain-gate.json`, `artifacts/sbom.json`, `artifacts/sbom.cyclonedx.json`, `artifacts/sbom.spdx.json`, and `artifacts/package-audit.json`; the setup supply-chain gate shape is locked separately by `docs/schemas/setup-supply-chain-gate.schema.json`. The internal SBOM artifact plus standard CycloneDX 1.5 JSON and SPDX 2.3 JSON SBOM artifacts are generated from `pnpm list --json --depth Infinity`, the audit artifact is generated from `pnpm audit --json`, dependency/lockfile file hashes are compared before/after install, `package.json` dependency sections are semantically diffed for added/removed/version-changed dependencies, `pnpm-lock.yaml` is semantically diffed for importer dependency additions/removals/specifier changes/resolved-version changes and package version changes, and critical/high audit findings block setup completion unless `--audit-exception` points to a project-local `.ai-web/approvals/*.json` file with `approved: true`, `accepted_risk: true`, matching blocked severities/package manager, finding-bound package/advisory coverage or matching audit artifact hash, non-future `approved_at`, unexpired `expires_at`, and a rollback plan. Nonzero `pnpm audit` exits fail closed unless the JSON output is recognized as an audit findings payload. Setup package-manager subprocesses run with lifecycle scripts disabled by default, the npmjs registry selected explicitly, and an allowlisted environment plus `unsetenv_others: true`, so sensitive parent variables such as `SECRET` and `NPM_TOKEN` are not inherited. `aiweb setup --install --approved --allow-lifecycle-scripts` is deliberately fail-closed: when lifecycle scripts are present it blocks before launching pnpm, records `lifecycle_enabled_requested`, `lifecycle_enabled_execution_available: false`, and a broker denial, and still requires a future sandbox driver plus OS/container egress-firewall evidence before scripts can execute. Setup now also enforces a dependency network allowlist at the package metadata boundary: direct `package.json` git/http/ssh/GitHub dependency specs and `pnpm-lock.yaml` network refs are scanned before launch and after install, token-like URL query parameters and URI userinfo are redacted before evidence persistence, and any host outside `registry.npmjs.org` blocks the setup run with `network_allowlist_enforcement` evidence. Post-install `package.json` missing/invalid/manifest-empty states and post-install `pnpm-lock.yaml` missing/invalid/unreadable states also block setup completion. aiweb itself does not read `.env*`; however, approved package-manager code still executes in the project root, and the allowlist is metadata/policy enforcement rather than an OS-level egress firewall. Any future lifecycle-enabled install mode would still need sandboxed file-access and network isolation before claiming lifecycle scripts cannot read workspace files or bypass egress policy.

Both engine-run and setup supply-chain gates now include `lifecycle_sandbox_gate` evidence. The gate records detected `preinstall`/`install`/`postinstall`/`prepare` scripts with redacted command text and command hashes, proves the default install path uses no lifecycle execution, and marks lifecycle-enabled install as `blocked_until_sandbox_and_egress_firewall` until container/VM isolation, deny-by-default egress logs, environment allowlist, `.env*` read denial, workspace-escape denial, SBOM, audit, and package-diff artifacts exist. Setup's default gate also records a `default_install_sandbox_attestation` showing `--ignore-scripts`, `unsetenv_others`, stripped secret parent env keys, no secret-value recording, and the explicit limitation that the host package-manager process is not an OS/container filesystem or egress sandbox. The explicit lifecycle-enabled flag is now a safe API surface, not execution permission: it records the request and denies it before pnpm when scripts are present. This is a truthful blocking contract, not a claim that the OS/container egress firewall is already implemented.

The local backend bridge that invokes `bin/aiweb` from HTTP routes now records its own `aiweb.backend.side_effect_broker` envelope before process launch. Real mutating backend bridge commands persist `.ai-web/runs/backend-bridge-*/side-effect-broker.jsonl` with `tool.requested -> policy.decision -> tool.started -> tool.finished|tool.failed|tool.blocked`; dry-run and read-only bridge commands keep inline broker evidence in the JSON response to avoid creating project files. This closes the previous backend-helper process-launch blind spot while preserving the stricter dedicated broker evidence for engine-run, OpenManus, deploy provider CLI, and approved setup install paths.

Lazyweb design research external HTTP calls also emit `aiweb.lazyweb.side_effect_broker` evidence before `Net::HTTP` starts. Real Lazyweb-backed design research writes `.ai-web/runs/lazyweb-research-*/side-effect-broker.jsonl` and includes the broker summary in the design-research payload; endpoint query tokens and bearer credentials are redacted before persistence. Dry-run design research remains plan-only and writes no broker file.

`aiweb mcp-broker call --server lazyweb --tool lazyweb_health|lazyweb_search`, `aiweb mcp-broker call --server project_files --tool project_file_metadata --query <relative-file>`, `aiweb mcp-broker call --server project_files --tool project_file_list --query <relative-file-or-directory> [--limit N]`, `aiweb mcp-broker call --server project_files --tool project_file_excerpt --query <relative-file> [--limit N]`, and `aiweb mcp-broker call --server project_files --tool project_file_search --query [<relative-file-or-directory>::]<literal-text> [--limit N]` are the approved implementation-worker MCP broker surfaces. Dry-run writes nothing. Unapproved calls write only deny/block audit evidence under `.ai-web/runs/mcp-broker-*/side-effect-broker.jsonl`; approved Lazyweb calls require configured Lazyweb credentials, restrict the server/tool/arguments, redact endpoint tokens and returned payloads, and persist `.ai-web/runs/mcp-broker-*/mcp-broker.json` plus `side-effect-broker.jsonl`. Approved `project_files.project_file_metadata` calls require a relative regular project file path, reject traversal, symlinks, `.env*`, `.git`, `node_modules`, and generated run artifacts, use no credentials or network destinations, and return only file metadata (`bytes`, `sha256`, `mtime_utc`) with `content_included: false`. Approved `project_files.project_file_list` calls require a relative safe file or directory path, reject traversal, symlinks, `.env*`, `.git`, `node_modules`, generated run artifacts, and secret-looking paths, skip unsafe child entries, enforce the shared `--limit 1..20` bound, use no credentials or network destinations, and return a sorted bounded metadata-only inventory with `content_included: false`. Approved `project_files.project_file_excerpt` calls require a relative regular UTF-8 text file, reject traversal, symlinks, `.env*`, `.git`, `node_modules`, generated run artifacts, secret-looking paths, oversized files, binary/non-UTF-8 content, and any content that would trigger secret redaction; they return at most `--limit 1..20` lines with `content_included: true`, no credentials, and no network destinations. Approved `project_files.project_file_search` calls require a relative regular UTF-8 text file or directory plus literal text, reject traversal, symlinks, `.env*`, `.git`, `node_modules`, generated run artifacts, secret-looking paths/patterns, oversized files, binary/non-UTF-8 content, and any candidate file that would trigger secret redaction; they scan at most 200 safe files and return at most `--limit 1..20` line excerpts with only a pattern hash, no credentials, and no network destinations. Unknown MCP/connectors also write a fail-closed `connector_policy` and broker log with `driver_status: missing_broker_driver_fail_closed`, plus the required fields a future driver must supply (`mcp_server`, tool allowlist, argument schema, credential source, delegated identity, network destinations, output redaction, per-call audit, broker path, result schema, and rollback/replay policy). This creates concrete implementation-worker MCP drivers for Lazyweb health/search and project file metadata/list/excerpt/search plus a machine-readable denial contract for all other connectors, but it is not a generic connector runner and is not available inside the default engine-run sandbox without explicit approval.

`aiweb eval-baseline review-pack|validate|import` is the human-calibrated eval baseline surface. Review packs contain placeholders only and explicitly forbid agent-filled scores or invented reviewer ids. Validation writes `.ai-web/eval/human-baseline-validation.json` with `corpus_readiness`: a corpus is `production_ready_multi_fixture` only when it has at least two calibrated fixtures, corpus metadata with reviewer count, at least two unique reviewer ids, and per-fixture `human_ratings` reviewer evidence. A single calibrated fixture may still be imported with `--approved` for local experimentation, but it is labeled `calibrated_but_not_production_corpus` and cannot be used as proof that the repository already owns a production human-calibrated multi-fixture corpus.

`--resume <run-id>` reads the prior checkpoint and staged manifest, reuses the prior staged workspace when present, records `run.resumed`, and copies back only after the same validation gate passes.

The checkpoint records a durable run graph cursor and artifact hashes for manifest, graph execution plan, graph scheduler state, sandbox preflight, authz enforcement, worker adapter registry, project index, run memory, verification, preview, browser evidence, design verdict, design fidelity, design fixture, eval benchmark, and diff artifacts that exist at checkpoint time. Graph-backed resume requires the mandatory artifact hash entries for `staged_manifest`, `graph_execution_plan`, `graph_scheduler_state`, `opendesign_contract`, `project_index`, `run_memory`, `authz_enforcement`, `worker_adapter_registry`, and `sandbox_preflight`; deleting only one required hash entry is treated as checkpoint tampering. The graph is replayable at the contract level: each node declares `node_id`, `node_type`, artifact inputs/outputs, state, attempt, retry policy, approval policy, side-effect boundary, executor descriptor, replay policy, idempotency key, and checkpoint cursor. The graph also carries an `executor_contract` with sequential node order, checkpoint policy, resume strategy, and side-effect gate. Before worker execution, aiweb writes `artifacts/graph-execution-plan.json`, staged `_aiweb/graph-execution-plan.json`, `artifacts/graph-scheduler-state.json`, and staged `_aiweb/graph-scheduler-state.json`. These artifacts are now produced by `Aiweb::GraphSchedulerRuntime` (`execution_driver: aiweb.graph_scheduler.runtime.v1`, `state_owner: graph_scheduler_runtime`) rather than by ad-hoc engine-run helper shape logic. The plan artifact is derived from `run_graph.executor_contract` and records `scheduler_type: sequential_durable_node_scheduler`, the selected `start_node_id`, node invocation handlers, replay policies, and validation booleans proving node order, handler presence, side-effect gating, artifact-hash replay requirements, and runtime-owned retry/replay/cursor/checkpoint evidence. The scheduler-state artifact records the resume start node, cursor, node states, attempts, output artifact refs, checkpoint ref, and append-only node transition records; it is checkpoint-hashed and emitted through `graph.scheduler.started`, `graph.node.finished`, and `graph.scheduler.finished` events. Completed runs update node states from the observed artifacts (`preflight`, `worker_act`, `verify`, `preview`, `observe_browser`, `design_gate`, `approval`, `copy_back`, `finalize`) so the graph is no longer just a static pending plan. Resume validates the graph cursor, executor contract, node executor descriptors, replay policies, required artifact hash keys/paths, the allowlisted artifact-hash key set, exact expected paths for every present known artifact hash, byte sizes, and artifact hashes before either continuing from an executable scheduler start node or fail-closing when the start node requires a dedicated continuation handler. The separate engine-scheduler tick/daemon service can now record and optionally execute the resume bridge from those artifacts, with foreground-loop heartbeat, queue ledger, duplicate-claim guard, and worker-pool slot evidence. Node bodies are still executed by registered engine handlers; the extracted runtime plus local service own scheduler state, lease, decision, transition, heartbeat, queue, and local worker-pool evidence, not a distributed background worker cluster.

When aiweb's sandbox verification fails and cycles remain, aiweb records `qa.failed`, writes `_aiweb/repair-observation.json` inside the staged workspace, records `repair.planned`, and invokes the worker again with that observation available. Copy-back still happens only after the final policy and verification pass.

If sandbox output, copy-back policy, or preflight evidence indicates secret leakage, raw environment disclosure, unexpected network/connector guard failure, host-root mutation, or a forbidden mount, the run enters `quarantined`. A quarantined run records a redacted quarantine summary alongside the normal redacted project-local evidence artifacts, records `run.quarantined`, blocks copy-back, and requires manual release outside `engine-run`.

## Guardrails

- No host root writable mount.
- Real `agentic_local` agent/tool execution must run through an aiweb-validated Docker/Podman no-network command.
- No `.env`, credentials, provider auth, browser profile, or secret-looking path access.
- Network is disabled by default.
- `openmanus:latest` stays a local-development default only. When `AIWEB_OPENMANUS_REQUIRE_DIGEST=1`, `AIWEB_REQUIRE_PINNED_OPENMANUS_IMAGE=1`, `AIWEB_ENGINE_RUN_STRICT_SANDBOX=1`, or an aiweb runtime environment variable such as `AIWEB_ENV=production` is set, OpenManus execution is blocked unless `AIWEB_OPENMANUS_IMAGE` is digest-pinned as `name@sha256:<digest>`.
- Package install, external network, deploy, provider CLI, MCP/connectors, raw environment reads, and git push are represented as structured blocked actions before copy-back and require explicit elevated approval.
- The engine-run supervised worker, verification, preview, and browser-observation flows go through the tool broker event flow: `tool.requested -> policy.decision -> tool.started -> tool.finished|tool.blocked`. Tool requests include tool name, redacted arguments/command descriptors, working directory, capability scope, risk class, expected outputs, idempotency key, and approval hash. In the staged OpenManus/OpenHands workspace, `_aiweb/tool-broker-bin` is prepended to container `PATH` so package-install, external-network, deploy/provider, `git push`, and raw-env-read commands are blocked before execution and mirrored into the run event stream as `tool.blocked`. The worker-adapter registry now fail-closes executable adapters that lack `broker_contract` coverage, records Lazyweb design-research, approved implementation-worker Lazyweb health/search, and approved `project_files.project_file_metadata` / `project_files.project_file_list` / `project_files.project_file_excerpt` / `project_files.project_file_search` as known brokered MCP drivers, and marks all other implementation-worker MCP/connectors, future adapters, and elevated runners as deny-by-default or broker-required surfaces. The approved `aiweb deploy --approved` provider CLI path also writes a per-run `.ai-web/runs/<deploy-run>/side-effect-broker.jsonl`, redacts provider stdout/stderr before persisting logs, records the broker event list in deploy metadata before/after provider CLI execution, and marks a failed provider CLI exit as a network/external side-effect attempt with unknown result rather than as no network attempt. The shared `ProjectSideEffectBroker` helper now owns broker plan/context/event writing plus command/output redaction so future backend helpers, adapters, and elevated runners can reuse the same contract instead of copying deploy-specific code. This covers the current engine-run/staged-worker, Lazyweb design-research MCP, approved Lazyweb implementation-worker MCP, approved project file metadata/list/excerpt/search MCP, and approved elevated paths, and fail-closes all other implementation-worker MCP plus known future adapter surfaces until broker drivers exist; it is still not an OS-level universal enforcement broker for arbitrary generated code.
- `sandbox-preflight.json` records generated argv, resolved executable path, parsed image inspect evidence (`RepoDigests`, image id, OS/architecture when available), whether the image reference is digest-pinned, preflight warnings for unpinned or unobservable evidence, runtime info from `docker|podman info` when available (rootless/rootful observation, security options, driver/cgroup hints), runtime container id via `--cidfile` plus fail-closed post-run `docker|podman inspect` HostConfig cross-check (network none, read-only rootfs, cap-drop ALL, no-new-privileges, non-root user, exactly one writable `/workspace` bind, and `/workspace` source matching the staged workspace), configured non-root `--user`, inside-container self-attestation (container hostname, effective uid/gid/user, env guards, workspace writability, read-only root check, `/proc/self/status` no-new-privs/seccomp/capability evidence, `/proc/self/cgroup` evidence, mountinfo excerpt, and Python socket egress-denial probe), optional required Docker/Podman runtime matrix verification (`AIWEB_ENGINE_RUN_RUNTIME_MATRIX=docker,podman` or `AIWEB_ENGINE_RUN_REQUIRE_RUNTIME_MATRIX=1`) that fail-closes on missing runtimes or unsupported matrix names, host/inside mounts, network mode, capabilities, security profile, resource limits, tmpfs, and negative checks proving project root, `.git`, `.env*`, cloud credentials, browser profiles, and host home directories are not mounted. Empty, malformed, or digestless image inspect output is blocking rather than advisory.
- CI runs `bin/engine-runtime-matrix-check --json` on push and pull request to build a disposable OpenManus smoke image under both Docker and Podman, execute engine-run with `AIWEB_ENGINE_RUN_RUNTIME_MATRIX=docker,podman`, and upload `.ai-web/ci/engine-runtime-matrix-smoke.json` as runtime-boundary evidence. This makes real runtime matrix evidence part of the normal CI contract instead of a manual-only smoke.
- Package install is never part of the default sandbox profile. Elevated install approval must name the package manager, exact command, registry allowlist, network allowlist, lifecycle-script policy, lockfile policy, expected changed files, timeout, rollback behavior, dependency diff, lockfile diff, package manager config, audit/SBOM output where available, and vulnerability copy-back gate; the run records this as `artifacts/supply-chain-gate.json`.
- MCP/connectors are denied by default for implementation workers except the approved `mcp-broker` Lazyweb health/search and `project_files.project_file_metadata` / `project_files.project_file_list` / `project_files.project_file_excerpt` / `project_files.project_file_search` paths. The Lazyweb design-research adapter is brokered by `aiweb.lazyweb.side_effect_broker` under `.ai-web/runs/lazyweb-research-*/side-effect-broker.jsonl` and is not exposed to implementation agents. Approved implementation-worker Lazyweb and project file metadata/list/excerpt/search calls are brokered by `aiweb.implementation_mcp_broker` under `.ai-web/runs/mcp-broker-*/side-effect-broker.jsonl` and `.ai-web/runs/mcp-broker-*/mcp-broker.json`; project file drivers use no network or credentials, with excerpt/search additionally bounded to safe UTF-8 non-secret content. Any other elevated implementation-worker MCP approval must still name the server, tool names, allowed argument schema, credential source, delegated identity, network destinations, output redaction policy, and per-call audit fields before that server can be used.
- Web Workbench is a later view/control layer, not a prerequisite for this runtime.
